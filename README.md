## Главные выводы

1.  **ASP.NET Core — мощный и гибкий фреймворк.** Позволяет быстро создавать веб-приложения и API благодаря встроенным возможностям, таким как Dependency Injection, конфигурация через `appsettings.json` и единая модель программирования.
2.  **Middleware — это основа обработки HTTP-запросов.** Они формируют конвейер, через который проходит каждый запрос. Порядок регистрации middleware критически важен: сначала идут глобальные обработчики (логирование, аутентификация), затем — маршрутизация.
3.  **Minimal API снижает порог входа.** Для создания простого веб-сервера не нужны контроллеры и сложная структура — достаточно одного файла `Program.cs` и нескольких строк кода с методами `MapGet` или `MapPost`.

## Ответы на вопросы

### 1. Чем Minimal API отличается от MVC в ASP.NET Core?

*   **Minimal API:** Предназначен для создания простых микросервисов и небольших приложений. Весь код размещается в `Program.cs`, нет контроллеров, минимальное количество зависимостей. Быстрее для изучения и запуска.
*   **MVC (Model-View-Controller):** Используется для больших приложений с пользовательским интерфейсом (Razor-страницы). Требует разделения на контроллеры, модели и представления, поддерживает мощные функции (фильтры, хелперы, валидация).

### 2. Что произойдёт, если поставить `app.MapGet(...)` до `app.Use(...)`?

Middleware, зарегистрированные **после** вызова `MapGet`, не будут обработаны для этого маршрута. Порядок регистрации важен, так как конвейер строится последовательно: если запрос находит свой обработчик (маршрут) раньше, чем доходит до `app.Use`, middleware из цепочки, расположенные после, не выполнятся.

### 3. Почему ASP.NET Core преобразует PascalCase в camelCase при сериализации JSON?

Это стандартное соглашение для JSON в JavaScript-среде, где принято использовать `camelCase` для имён свойств. ASP.NET Core использует систему `System.Text.Json`, которая по умолчанию настроена на этот формат для обеспечения совместимости с клиентским кодом на JavaScript/TypeScript. Это поведение можно переопределить через настройки `JsonSerializerOptions`.

### 4. Что означает код ответа HTTP 401? А 404? А 200?

*   **200 OK:** Запрос успешно обработан, сервер вернул ожидаемые данные.
*   **401 Unauthorized:** Запрос требует аутентификации. Пользователь не предоставил учётные данные или они неверны (например, отсутствует ключ `?key=secret`).
*   **404 Not Found:** Запрашиваемый ресурс не существует на сервере (неправильный URL).

### 5. Чем `dotnet run` отличается от `dotnet watch run`?

*   `dotnet run` — однократно собирает и запускает приложение. При изменении кода нужно остановить сервер (Ctrl+C) и запустить снова.
*   `dotnet watch run` — запускает "наблюдатель" (file watcher), который отслеживает изменения в файлах проекта. При сохранении изменений он автоматически перезапускает сервер, что ускоряет разработку (режим горячей перезагрузки).
